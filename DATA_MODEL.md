# Agora — Data Model

> **Version:** 2.0 | **Status:** AUTHORITATIVE

---

## Entity-Relationship Diagram (RAP Business Object Hierarchy)

```
ZAGORA_LOCATION (root BO)
  └── ZAGORA_VISIT (child BO, composition)

ZAGORA_ADMIN_USER (standalone root BO — not composed under Location)
```

All entities use `sysuuid_x16` primary keys, generated by the RAP framework on Create (`%key` auto-fill in managed scenarios).

---

## ABAP Domain Definitions

Two fixed-value domains replace the PostgreSQL `ENUM` types. Both are referenced in CDS views and behavior definitions.

| Domain | Values | Description |
|---|---|---|
| `ZAGORA_D_LOC_TYPE` | `RESTAURANT`, `COFFEEHOUSE`, `BAR`, `BAKERY` | Location type classification |
| `ZAGORA_D_STATUS` | `VISITED`, `WISHLIST` | Lifecycle status; default value is `WISHLIST` |

### Overall Score Computation

`overall_score = (price_rating + ambience_rating + quality_rating) / 3.0`

- Computed in **application code** inside the `MODIFY` method of `ZAGORA_BP_LOCATION`, not as a HANA generated column.
- **ABAP null handling:** ABAP's initial value for numeric types is `0`, not NULL. The behavior implementation must explicitly check whether each rating field is at its initial value before computing the score. If any of the three rating fields is unset (initial value), `OverallScore` is set to null in the HANA column (the column allows NULL; the ABAP statement assigns the null explicitly). Do **not** rely on ABAP arithmetic with initial values producing a meaningful result.
- Returns null if any dimension is unset — including for wish-list entries that typically have no ratings.
- Recalculated and stored on every `MODIFY` operation that includes `PriceRating`, `AmbienceRating`, or `QualityRating` in the changed field set.
- The Archive "highest ranked" stat (`GET GetDashboardStats()`) excludes locations where `OverallScore IS NULL`.

## `ZAGORA_VISIT` (Child BO — Composition under Location)

Physical HANA transparent table. Compound key `(LocationUuid, VisitUuid)`.

| Field | ABAP Type | Key | Notes |
|---|---|---|---|
| `LocationUuid` | `sysuuid_x16` | Yes | FK to `ZAGORA_LOCATION`; part of compound key |
| `VisitUuid` | `sysuuid_x16` | Yes | Generated by RAP framework on Create |
| `VisitedAt` | `abap.dats` | | NOT NULL; the calendar date of the visit |
| `Note` | `abap.string(0)` | | nullable |
| `AdminUserId` | `abap.char(256)` | | Stores the BTP IAS principal name of the admin who logged the visit. Set by the RAP behavior implementation using `cl_abap_context_info=>get_user_technical_name()` on Create. **Never exposed in the public CDS projection** `ZAGORA_C_VISIT_PUB`. |
| `CreatedAt` | `abap.utclong` | | RAP managed |
| `LastChangedAt` | `abap.utclong` | | RAP managed; ETag master for this child entity |

### AdminUserId Privacy

`AdminUserId` is defined on the **interface** CDS view `ZAGORA_I_VISIT` only. It is structurally excluded from the public consumption view `ZAGORA_C_VISIT_PUB` — the field is not projected. This provides defense-in-depth: even if an unauthorized request reaches the public service binding, the field is architecturally absent from the OData response.

The admin consumption view `ZAGORA_C_VISIT` may include `AdminUserId` joined with `DisplayName` from `ZAGORA_ADMIN_USER` for attribution display in the admin UI.

## `ZAGORA_ADMIN_USER` (Standalone Root BO)

Physical HANA transparent table. This is not a transactional BO in the location-management sense — it is a user roster that an admin manages via Fiori.

| Field | ABAP Type | Key | Notes |
|---|---|---|---|
| `UserUuid` | `sysuuid_x16` | Yes | RAP-managed on Create |
| `IasPrincipal` | `abap.char(256)` | | IAS user name / email; UNIQUE constraint; the external identity used to match the acting user |
| `DisplayName` | `abap.char(100)` | | NOT NULL; human-readable name shown in the admin UI |
| `CreatedAt` | `abap.utclong` | | RAP managed |
| `LastChangedAt` | `abap.utclong` | | RAP managed |

**Relationship to BTP IAM:** Role enforcement (whether a given IAS principal can perform write operations) is done via BTP role collections assigned in the BTP cockpit, **not** via a column in this table. This table exists so the admin UI can display the user roster and so the behavior implementation can enforce the "cannot self-delete" rule by comparing the acting user's principal against `IasPrincipal`.

## Removed from Prior Version (v1.0 → v2.0)

| Removed Object | Reason |
|---|---|
| `images` table | Image gallery feature dropped; no file storage layer in v2 |
| `refresh_tokens` table | JWT rotation replaced by SAP BTP IAS session management |
| `lat`, `lng` columns on `locations` | Nominatim geocoding and interactive map (The Atlas) dropped |
| `ig_token_enc`, `fb_token_enc` columns on `admin_users` | Social sharing (Instagram / Facebook) dropped |
| `password_hash` column | Password management delegated to IAS; no application-layer bcrypt |
